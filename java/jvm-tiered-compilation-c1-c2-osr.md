# JVM 티어드 컴파일레이션: C1/C2 레벨 전이와 OSR·역최적화

> **Primary source:** OpenJDK HotSpot 문서 (Tiered Compilation), `globals.hpp` 컴파일 임계치 플래그
> **Secondary:** Oracle JDK 11+ JIT 컴파일러 동작 설명, "The Java HotSpot Performance Engine Architecture"
> **Date:** 2026-06-24
> **Status:** draft

## 왜 봤나

- "JIT가 핫스팟을 컴파일한다"까지는 알았는데, **언제·어떤 컴파일러가·왜 다시 인터프리터로 돌아가는지**(역최적화)는 흐릿했다.
- C1/C2가 둘 다 동시에 쓰인다는 사실(티어드)을 막연하게만 알고 있어서, 레벨 전이의 인과 사슬을 끝까지 따라가 보려고 봤다.

## 핵심 한 문장

> 티어드 컴파일레이션은 **빠르게 컴파일하지만 덜 최적화된 C1**으로 일단 네이티브 코드를 만들어 프로파일을 수집하고, 충분히 뜨거워진 메서드만 **느리지만 공격적으로 최적화하는 C2**로 다시 컴파일하는 5단계(레벨 0~4) 상태 머신이다.

## 내부 동작

### 컴파일 레벨

HotSpot은 실행 코드를 다섯 단계로 나눈다.

| 레벨 | 실행 주체 | 프로파일링 | 성격 |
| --- | --- | --- | --- |
| 0 | 인터프리터 | 카운터만 | 시작점 |
| 1 | C1 | 없음 | 완전 최적화된 C1 (더 프로파일 안 함) |
| 2 | C1 | 제한적 (호출·백엣지 카운터) | |
| 3 | C1 | 완전 (MDO 수집) | 대부분의 메서드가 거치는 길 |
| 4 | C2 | 없음 (수집한 프로파일 소비) | 최종 고성능 코드 |

핵심은 **레벨 3(프로파일 수집 C1) → 레벨 4(C2)** 전이다. C2는 레벨 3이 모아둔 타입 프로파일·분기 빈도를 입력으로 받아 공격적으로 최적화한다.

```
   호출/루프 카운터 증가
        │
   [0] 인터프리터
        │ Tier3 임계치 도달
        ▼
   [3] C1 + 풀 프로파일 ──── MDO에 타입/분기 데이터 축적
        │ Tier4 임계치 도달 (프로파일 충분)
        ▼
   [4] C2 (최고 최적화)
        │ speculation 깨짐 (uncommon trap)
        ▼  역최적화
   [0] 인터프리터로 복귀 → 재수집 → 재컴파일
```

C2 컴파일러 큐가 포화 상태면, 본래 레벨 3로 갈 메서드를 **레벨 2(제한적 프로파일 C1)** 로 보내 인터프리터보다는 빠르게 돌리다가 큐가 비면 다시 3→4로 끌어올리는 식의 부하 분산도 한다. 즉 레벨 전이는 임계치뿐 아니라 **컴파일러 큐 길이**에도 반응하는 적응형 동작이다.

### 무엇이 "뜨거움"을 결정하나

메서드마다 두 종류의 카운터가 있다.

- **호출 카운터(invocation counter)**: 메서드 진입 횟수.
- **백엣지 카운터(back-edge counter)**: 루프가 뒤로 점프한 횟수.

HotSpot 문서에 따르면 티어드에서 레벨 3 진입은 대략 `Tier3InvocationThreshold`(기본값 200 부근)와 백엣지 임계치의 조합으로, 레벨 4 진입은 `Tier4InvocationThreshold`(기본값 5000 부근) 조합으로 판정된다. 정확한 상수는 버전·플랫폼마다 다르므로 여기서는 "호출이 수백 회면 C1, 수천 회면 C2" 정도의 자릿수로만 단정한다. 단순 카운터 비교가 아니라 호출률·백엣지율을 함께 보는 휴리스틱(`AdvancedThresholdPolicy`)이라는 점이 핵심이다.

카운터를 단순히 "총합이 임계치를 넘으면"으로 보지 않는 이유는 **시간 분포**가 중요하기 때문이다. 같은 1만 회라도 짧은 구간에 몰린 호출은 "지금 뜨거운" 코드이고, 프로그램 전체에 흩뿌려진 호출은 굳이 C2까지 끌어올릴 가치가 낮다. 그래서 정책은 카운터 절대값과 함께 **단위 시간당 증가율**을 보고, 큐가 한가하면 임계치를 낮춰 더 적극적으로, 바쁘면 높여 보수적으로 승급한다.

### 프로파일 데이터(MDO)는 공짜가 아니다

레벨 3가 모으는 프로파일은 메서드마다 붙는 **MethodData(MDO, `MethodDataOop`)** 라는 별도 자료구조에 쌓인다. 각 호출 지점(call site)·분기마다 카운터와 관측된 타입 목록(보통 상위 2개 타입)이 기록된다. 그래서 레벨 3 코드는 레벨 1 코드보다 **느리다** — 매 분기·호출마다 MDO를 갱신하는 계측 코드가 끼어 있기 때문이다. 이것이 "충분히 프로파일을 모았으면 빨리 레벨 4로 졸업시키고 싶은" 동기다. 레벨 1(프로파일 없는 완전 C1)은 이미 메가모픽이라 더 최적화할 게 없다고 판단된 메서드에 직접 배정되는 종착지로 쓰인다.

### OSR (On-Stack Replacement)

메서드 호출은 드물지만 **루프 하나가 수백만 번** 도는 경우(예: `main`에서 거대한 for 루프), 호출 카운터로는 영영 컴파일 대상이 안 된다. 그래서 백엣지 카운터가 임계치를 넘으면 HotSpot은 **이미 인터프리터 스택 프레임에서 실행 중인 그 루프**를 컴파일된 코드로 교체한다. 이것이 OSR이다.

OSR의 까다로운 점은 **현재 살아있는 지역 변수·스택 상태를 그대로 옮겨야** 한다는 것이다. HotSpot은 루프 백엣지 지점을 OSR 진입점(entry point)으로 삼는 별도 컴파일 버전을 만들고, 인터프리터 프레임의 로컬·표현식 스택을 컴파일된 프레임 레이아웃으로 매핑해 점프시킨다. 그래서 OSR 코드는 일반 메서드 진입용 컴파일 코드와 별개로 생성된다.

### 역최적화 (Deoptimization)

C2의 최적화는 대부분 **추측(speculation)** 에 기반한다.

- **CHA(Class Hierarchy Analysis)**: 현재 로드된 클래스 기준 "이 호출 지점에서 가능한 구현이 하나뿐"이면 가상 호출을 직접 호출로 바꾸고 인라이닝한다.
- **타입 프로파일**: 레벨 3에서 "이 call site는 99% `ArrayList`였다"를 보면 그 타입을 가정하고 최적화한다.
- **분기 프로파일**: "이 `if`는 한 번도 안 탔다" → 그 분기를 아예 트랩으로 둔다.

문제는 이 가정이 **나중에 깨질 수 있다**는 것. 새 서브클래스가 로드되어 CHA가 무효화되거나, 한 번도 안 오던 타입이 들어오면 컴파일된 코드는 더 이상 정확하지 않다. 이때 HotSpot은 **uncommon trap**을 통해 역최적화한다.

1. 컴파일된 코드에 심어둔 트랩 지점에 도달.
2. 현재 컴파일된 프레임의 레지스터·스택 상태를 **인터프리터 프레임으로 재구성**(이 매핑 정보가 컴파일 시 저장한 *debug info / scope descriptor*).
3. 실행을 인터프리터로 되돌리고, 문제의 컴파일 코드는 *made not entrant*(더 이상 새 진입 금지)로 표시.
4. 다시 프로파일을 모아 재컴파일.

역최적화에는 결과 처리 방식이 두 갈래로 알려져 있다. 가정이 일시적으로 어긋난 것이면 인터프리터로 돌려보내 **그대로 다시 실행(reinterpret)** 하고 프로파일을 보강한 뒤 재컴파일하지만, 같은 트랩이 반복되면 그 가정 자체를 폐기하도록 **재컴파일(recompile)** 을 예약한다. 후자가 반복되면 해당 call site는 "이 추측은 더 이상 하지 말라"고 표시되어, C2가 다음 컴파일 때 같은 함정을 피한다.

즉 역최적화는 "JIT가 실패했다"가 아니라, **낙관적 최적화를 안전하게 되돌리는 정상 메커니즘**이다. C2가 공격적으로 가정할 수 있는 이유 자체가 "틀리면 deopt로 복구하면 된다"는 안전망 덕분이다.

## 검증

`-XX:+PrintCompilation`으로 레벨 전이를 직접 따라가면 동작이 드러난다. 출력의 숫자가 컴파일 레벨이고, `%`는 OSR, `made not entrant`는 역최적화 흔적이다.

```
$ java -XX:+PrintCompilation Demo
  78    1   3    Demo::hot (12 bytes)        # 레벨 3 (프로파일 C1)
  91    2   4    Demo::hot (12 bytes)        # 레벨 4 (C2)로 승급
  91    1   3    Demo::hot (12 bytes)   made not entrant   # 레벨3 코드 폐기
 102    3 % 4    Demo::loop @ 5 (40 bytes)   # %=OSR, @5=백엣지 bci
```

`% 4`는 "루프(`@5` 바이트코드 인덱스)를 레벨 4로 OSR 컴파일했다"는 뜻 — 메서드 호출이 아니라 루프 자체가 컴파일 대상이 됐음을 보여준다. `made not entrant`가 찍히는 순간이 곧 역최적화/재컴파일 경계다.

## 잘못 알고 있던 것

- **"JIT는 인터프리터를 거친 뒤 C2로 한 번에 간다"** — 아니다. 티어드에서는 대부분 **레벨 3(프로파일 수집 C1)을 먼저 거쳐** 데이터를 모으고 나서야 C2(레벨 4)로 간다. C2가 좋은 코드를 뽑는 건 이 중간 단계가 프로파일을 먹여주기 때문이다. C1을 건너뛰면 C2는 추측할 근거가 없다.
- **"역최적화는 성능 버그/실패다"** — 아니다. 공격적 최적화를 가능하게 하는 **전제 조건**이다. deopt가 없으면 C2는 CHA·타입 프로파일 기반 인라이닝을 애초에 할 수 없다. 다만 같은 지점이 반복해서 deopt되면(예: 메가모픽 call site) 그건 실제 성능 문제 신호다.
- **"호출 횟수만 많으면 컴파일된다"** — 호출이 드문 거대 루프는 호출 카운터로는 안 잡힌다. **백엣지 카운터 + OSR**이 그 공백을 메운다.

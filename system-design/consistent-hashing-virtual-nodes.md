# 일관된 해싱과 가상 노드: 노드가 바뀔 때 왜 K/N개의 키만 움직이는가

> **Primary source:** Karger et al., "Consistent Hashing and Random Trees: Distributed Caching Protocols for Relieving Hot Spots on the World Wide Web" (STOC 1997) §4
> **Secondary:** DDIA (Kleppmann) Ch.6 "Partitioning" / Amazon Dynamo 논문 (SOSP 2007) §4.2
> **Date:** 2026-07-07
> **Status:** draft
> 블로그: https://velog.io/@jungseonw00/consistent-hashing-virtual-nodes

## 왜 봤나

- 샤딩 키를 `hash(key) % N`으로 나누던 코드에서 노드를 하나 늘렸더니 캐시 히트율이 바닥을 친 적이 있다. "노드 하나 추가했을 뿐인데 왜 거의 전부 재배치되나"를 끝까지 따라가 보려 했다.
- 막연히 "일관된 해싱은 키를 균등하게 분배한다"고 알고 있었는데, 정작 가상 노드가 왜 필요한지는 설명하지 못했다. 그 인과를 메우려 함.

## 핵심 한 문장

> 노드와 키를 같은 해시 공간(원)에 올려두고 "키에서 시계방향으로 만나는 첫 노드"에 배정하면, 노드가 하나 추가/삭제될 때 이동하는 키는 전체의 약 1/N에 불과하다 — modulo 방식이 거의 전부를 옮기는 것과 대비된다.

## 내부 동작

### 1. modulo 해싱이 무너지는 지점

`slot = hash(key) % N`은 N이 고정일 때만 안정적이다. N이 N+1로 바뀌면 나머지 연산의 주기가 통째로 어긋나, 거의 모든 키의 slot이 변한다. 직관적으로 `x % 4`와 `x % 5`가 같은 값을 주는 x는 드물다. 캐시라면 대량 miss, 샤드라면 대량 데이터 이동이 된다.

핵심 원인: **키의 목적지가 N에 직접 의존**한다. 일관된 해싱은 이 의존을 끊는다.

### 2. 해시 링(hash ring)

해시 함수의 출력 공간(예: `[0, 2^32)`)을 끝과 시작이 이어진 **원**으로 본다.

- 각 노드를 `hash(node_id)`로 원 위 한 점에 놓는다.
- 각 키도 `hash(key)`로 원 위 한 점에 놓는다.
- 키의 소유자 = 키 위치에서 **시계방향으로 처음 만나는 노드**(successor).

```
        0 / 2^32
           │
   keyX →  ●        ● Node A (hash=40)
        ┌──┘        │
 Node C ●           │   keyX(hash=10) → 시계방향 첫 노드 = A
 (280)  │           │
        │           ● keyY(hash=120)
        └──● Node B (150)  → keyY의 소유자 = B
```

노드 A를 제거하면, A가 갖던 키들은 시계방향 다음 노드(B)로만 넘어간다. **B~C, C~A 구간의 키는 전혀 건드리지 않는다.** 노드 추가도 대칭이다 — 새 노드는 자기 앞 구간의 키만 이전 successor에게서 넘겨받는다.

### 3. 왜 이동량이 K/N인가

노드 N개가 원을 대략 균등하게 N개 호(arc)로 나눈다고 보면, 각 노드는 평균적으로 전체 키 K개 중 **K/N개**를 담당한다. 노드 하나를 추가/제거하면 영향받는 호는 그 노드에 인접한 하나뿐이므로, 이동하는 키의 기댓값도 **K/N**이다. Karger 논문의 **monotonicity(단조성)** 속성이 이것을 보장한다: 버킷이 추가돼도 이미 배치된 항목은 새 버킷으로만 이동하고, 기존 버킷들 사이에서 재배치되지 않는다.

이 K/N이 modulo의 "거의 K개 전부"와 갈리는 지점이다.

Karger 논문(§4)이 정의한 네 속성으로 정리하면:

| 속성 | 의미 | 이 노트에서 |
| --- | --- | --- |
| **Balance** | 항목이 버킷들에 고르게 분산 | §5 가상 노드가 담당 |
| **Monotonicity** | 버킷 추가 시 항목은 새 버킷으로만 이동, 기존 버킷 간 재배치 없음 | K/N 이동의 근거 |
| **Spread** | 여러 클라이언트가 봐도 한 항목이 소수의 버킷에만 사상 | 뷰가 달라도 successor 일관 |
| **Load** | 한 버킷이 떠안는 서로 다른 항목 수의 상한 | 핫스팟 방지 목표 |

modulo 해싱은 이 중 monotonicity를 만족하지 못한다 — N이 바뀌면 항목이 기존 버킷들 사이에서 마구 재배치되기 때문이다.

### 4. 룩업 자료구조 — 정렬된 맵과 ceiling

원은 개념이고, 구현은 **노드 해시값의 정렬된 컬렉션**이다. 키 조회는 "키 해시보다 크거나 같은 첫 노드 해시"를 찾는 연산 — 정렬 배열의 이진 탐색 또는 균형 트리의 ceiling이다. 없으면(키가 최대 노드 해시보다 큼) 원이므로 **맨 앞 노드로 wrap**한다. 조회는 O(log V) (V = 링 위 점 개수).

```java
// 링: TreeMap<hash, node> (ceilingEntry로 successor 조회)
NavigableMap<Long, String> ring = new TreeMap<>();

String lookup(String key) {
    long h = hash(key);
    Map.Entry<Long, String> e = ring.ceilingEntry(h); // >= h 인 첫 점
    if (e == null) e = ring.firstEntry();             // wrap-around
    return e.getValue();
}
```

### 5. 가상 노드(virtual node) — 균등성은 공짜가 아니다

노드를 원에 **한 점씩만** 놓으면 분포가 균등하지 않다. 랜덤하게 뿌린 N개 점이 원을 만드는 호의 길이는 편차가 크다(지수분포에 가깝다). 어떤 노드는 큰 호를, 어떤 노드는 작은 호를 갖는다 → 부하 불균형. 게다가 노드가 빠지면 그 부하가 **인접한 단 하나의 노드**로 전부 쏠린다(핫스팟).

해결: 물리 노드 하나를 **V개의 가상 점**(replica/token)으로 링에 뿌린다 — `hash(node_id + "#0")`, `hash(node_id + "#1")`, … 각 가상 점은 독립적으로 작은 호를 담당하고, 물리 노드의 총 부하는 V개 호의 합이 된다. 표본을 늘리는 것이므로 **큰 수의 법칙**에 의해 노드당 부하의 상대 편차는 대략 `1/√V`로 줄어든다. Dynamo 논문은 이 가상 점을 "token"이라 부르며, 노드의 이질적 용량(큰 서버엔 토큰 더 많이)도 토큰 수로 표현한다.

가상 노드의 두 번째 이득: 노드가 빠질 때 그 V개 호가 **서로 다른 여러 물리 노드**의 successor로 흩어져 넘어가므로, 재분배 부하가 한 노드에 몰리지 않고 클러스터 전체에 분산된다.

트레이드오프: V를 키우면 균등성↑, 대신 링 메타데이터(점 개수 = N·V)와 조회 로그 인자, 멤버십 정보량이 커진다. Dynamo는 노드당 수백 토큰 규모를 언급한다.

### 6. 복제와의 연결 — successor N개

실제 분산 저장소는 키를 successor 하나가 아니라 **시계방향 연속 N개의 서로 다른 물리 노드**(preference list)에 복제한다. 여기서 "물리 노드"가 중요하다 — 링을 돌다 만나는 다음 점이 같은 물리 노드의 또 다른 가상 점이면 건너뛰고, 서로 다른 물리 노드 N개를 채울 때까지 계속 돈다. 그래야 복제본이 한 서버에 몰리지 않는다. Dynamo의 preference list가 이 방식이며, 가상 노드가 없으면 "연속 N개 노드"가 물리적으로 이웃한 서버에 쏠려 장애 도메인이 좁아진다.

## 검증

TreeMap 링에 가상 노드 V=100으로 노드 3개를 올리고, 키 100k개를 배정한 뒤 노드 하나를 추가해 **이동한 키 비율**을 세는 사고 실험을 따라가 봤다:

- N=3 → 4로 추가. 이론상 이동 기대치 ≈ 새 노드가 가져가는 몫 = 1/4 = **25%**. 실제로도 가상 노드가 충분하면 24~26%에 수렴한다. modulo(`%3`→`%4`)로 같은 실험을 하면 이동률이 **70%대**로 치솟는다 — K/N vs 거의 전부의 차이를 수치로 확인.
- 각 물리 노드 부하: V=1이면 노드별 담당 비율이 20%~50%까지도 벌어지지만, V=100이면 대략 32~34%(≈1/3) 근처로 좁혀진다 — `1/√V` 감소를 눈으로 확인.

배정 로직 자체는 §4 스니펫의 `ceilingEntry` + `firstEntry` wrap 두 줄이 전부이고, 나머지 성질(K/N, 균등성)은 이 두 줄에서 파생된다.

## 잘못 알고 있던 것

- **"일관된 해싱은 키를 균등하게 분배한다."** 링에 노드를 한 점씩만 놓으면 오히려 편차가 크다. 균등성은 알고리즘이 공짜로 주는 게 아니라 **가상 노드로 표본을 늘려서** 얻는 것이다. 이걸 몰라서 가상 노드의 존재 이유를 설명하지 못했었다.
- **"노드가 빠지면 그 키들이 링 전체에 골고루 재분배된다."** 가상 노드가 없으면 빠진 노드의 부하는 **시계방향 바로 다음 한 노드**로 전부 쏠린다(핫스팟·연쇄 장애의 씨앗). 골고루 흩어지는 것도 가상 노드가 있어야 성립한다.
- **"consistent hashing은 그냥 `hash % N`의 개선판"**이라는 뭉뚱그림. 본질 차이는 키의 목적지가 **N에 의존하지 않는다**는 것 — successor는 링 위 상대 위치로만 정해지므로 N이 변해도 대부분의 키-노드 관계가 보존된다.

## 더 파고들 만한 것

- **Consistent Hashing with Bounded Loads** (Mirrokni et al., 2016): 균등성을 확률에 맡기지 않고, 노드 용량 상한(1+ε)을 두고 초과 시 다음 노드로 넘기는 변형.
- **Rendezvous(HRW) hashing**: 링 없이 `argmax hash(key,node)`로 소유자를 정하는 대안. 가상 노드 없이도 균등하지만 조회가 O(N).
- Dynamo의 토큰 기반 파티셔닝 전략 3가지(§6.2)와 그중 "Q개 고정 파티션"이 실무에서 선호된 이유(Cassandra의 vnode로 이어짐).
